# AGENTS.md

## Project overview

Oolong is an ephemeral, keyboard-driven terminal chat client written in Go. It supports OpenAI, Anthropic, Ollama, and OpenAI-compatible endpoints through a Bubble Tea TUI and a non-interactive one-shot mode.

Preserve the project's core principles:

- Conversations remain in memory unless the user explicitly saves a transcript.
- Never send test traffic to real provider APIs or read/write real credentials.
- Keep the interface keyboard-first, compact, and usable across macOS, Linux, and Windows.
- A malformed or unavailable optional configuration should not prevent startup.

## Repository map

- `main.go`: CLI flags, configuration loading, provider setup, and TUI/one-shot entry points.
- `internal/ui/`: Bubble Tea state, picker, chat, streaming, transcript/resume, key manager, and rendering.
- `internal/openai/`, `internal/anthropic/`, `internal/ollama/`: provider clients.
- `internal/config/`: TOML configuration and built-in model catalog.
- `internal/oneshot/`: stdin/argument-driven streaming mode.
- `internal/keystore/`: OS keychain integration.
- `internal/clipboard/`: clipboard support, including the non-cgo fallback.
- `internal/mathfmt/`: LaTeX-to-Unicode formatting.
- `e2e/`: fake API and scripted PTY smoke tests.
- `demo/`: deterministic VHS demo assets.
- `PLAN.md`: product priorities and explicit non-goals; consult it before adding features.

## Development guidelines

- Follow idiomatic Go and run `gofmt` on changed Go files.
- Keep provider-specific behavior behind the existing client boundaries. Preserve shared behavior across TUI and one-shot paths.
- Treat the UI as a state machine. Update commands, messages, model state, and views consistently, and add focused tests for transitions.
- Preserve cancellation and streaming semantics. Do not block the Bubble Tea update loop.
- Keep user-visible copy concise and terminal-width aware.
- Maintain cross-platform behavior. Do not assume a Unix shell, filesystem layout, or keychain implementation in Go code.
- Do not persist chats, attachments, prompts, or credentials implicitly. OpenAI requests must continue to disable server-side response storage.
- Update `README.md` when changing flags, configuration, keybindings, provider behavior, or other user-facing features.
- Update `PLAN.md` only when the roadmap or standing product decisions genuinely change.

## Testing and verification

For ordinary changes, run:

```sh
gofmt -w <changed-go-files>
go test ./...
go vet ./...
go build ./...
```

Add or update adjacent `*_test.go` coverage for behavioral changes. Prefer table-driven tests when several inputs share the same contract.

For changes affecting startup, provider routing, model selection, streaming, terminal input, transcripts, attachments, or one-shot mode, also run:

```sh
./e2e/smoke.sh
```

The E2E harness uses `e2e/fakeapi`; reuse it instead of contacting a real API. When writing a custom PTY scenario, follow the environment isolation and terminal-query handling documented in `.claude/skills/verify/SKILL.md`.

Linux builds with cgo require X11 development headers for `golang.design/x/clipboard`. The project must still build with `CGO_ENABLED=0`, with clipboard image paste disabled through the existing fallback.

## Commits and releases

Use conventional commit subjects because pushes to `main` drive automated releases:

- `feat:` triggers a minor release.
- `fix:` triggers a patch release.
- `!` or a `BREAKING CHANGE` footer triggers a major release.
- `chore:`, `docs:`, and `test:` do not trigger a release.

Do not manually edit generated release artifacts. The demo GIF is regenerated by its workflow from `demo/demo.tape` and the fake API.
